# ADR-003: dads:addressField をカスタムコンポーネントで実装する

Date: 2026-05-29
Status: Accepted

## Context

郵便番号検索＋住所入力エリアを再利用可能なコンポーネントとして切り出す際、
JSF の実装方式として複合コンポーネント（XHTML ベース）とカスタムコンポーネント（Java ベース）の2択を検討した。

他の DADS コンポーネント（`dads:inputField`、`dads:button` 等）は複合コンポーネントで実装している。
`dads:addressField` は郵便番号の複数候補時にモーダルを表示する要件を持ち、
Ajax をまたいだ状態管理（候補リストの保持）が必要になる点が他コンポーネントと異なる。

## Decision

**`dads:addressField` はカスタムコンポーネント（`UIInput` 継承 + `Renderer`）で実装する。**

目的は2つ。
1. 複合コンポーネントとカスタムコンポーネントの良し悪しをこのプロジェクト内で比較・検証する
2. モーダル表示と候補リストの状態管理を `saveState` / `restoreState` でコンポーネント自身に閉じ込める

## Consequences

**良い点**
- `UIComponentBase` の状態保存機構で候補リストをコンポーネント内に保持でき、補助 Bean が不要
- `Renderer` で HTML を完全制御できるため、モーダルの構造や CSS を自由に生成できる
- 複合コンポーネントとの実装上の違いをこのプロジェクトで直接体験できる

**トレードオフ**
- 複合コンポーネントと比べて実装量が多い（Component クラス + Renderer クラス + タグライブラリ登録）
- XHTML を読むだけではレンダリング内容が把握できず、Java コードを追う必要がある
